# Spike 4 — tree -> terminal end-to-end via the Watch dirty set

De-risks genesis **A4**: a live human-facing TUI backend driven off the SAME
perception element tree, repainting only what changed.

## Question

Can a live perception tree containing Watch-driven state render to spike 2's
double-buffered cell grid such that a stream event flows:

```
event -> perceived() -> owner dirty set -> onNeedsHarvest
      -> flushHarvest -> targeted repaint -> minimal ANSI diff
```

with repaint **locality** (changed cells confined to the changed node's screen
region)?

## Answer: YES — proven end-to-end

- `LiveLoop` mounts the tree with `PerceptionOwner.mountRoot`, hooks
  `owner.onNeedsHarvest` to `scheduleMicrotask(_flushPass)`, and each pass does
  `flushHarvest()` -> repaint ONLY the rebuilt boxes into the back buffer ->
  `swap()` -> `AnsiEncoder.encodeBytes(changes)`. Spike 2's grid/diff/encoder
  is used as-is (path dep), spike 3's `Field` is the leaf vocabulary.
- One stream event costs **exactly one flush pass** (asserted), and the
  resulting cell diff is non-empty, lies **entirely inside the watched box's
  rect**, and touches **zero cells** in the static box's rect and the other
  live box's rect — the A4 locality crux (test b).
- An event whose value renders identically still rebuilds the Watch and
  repaints its box into the back buffer, but the double-buffer diff dedups it
  to **0 changed cells / 0 ANSI bytes** (test c; demo frame 6).
- The static box's element instance is never rebuilt: its build counter stays
  at 1 (mount only) across all events, and the element identity is unchanged
  (test e). Watch builder call counts match events exactly (mount + N).
- Multiple events arriving before a flush coalesce naturally: `onNeedsHarvest`
  fires only on the dirty set's empty -> non-empty edge, so one microtask pass
  drains them all (observed, not separately asserted).

## Per-frame economy (recorded numbers)

Grid 40x12 = **480 cells**; full-redraw baseline (same encoder over every
cell) = **1053 bytes/frame**.

Test (d), 5 ticker events + 1 feed event:
per-frame changed cells `[2, 2, 2, 6, 7, 6]`,
per-frame ANSI bytes `[24, 24, 24, 28, 29, 21]` — every update frame under
1.5% of the grid and under 3% of a full redraw.

Demo (10 scripted events): frame 0 full scene = 297 cells / 979 bytes; the 10
update frames totalled **268 bytes** (`0..38` bytes each, duplicate event = 0)
vs 10530 bytes for 10 full redraws — **~39x cheaper**, and that's with a tiny
scene; locality makes the ratio scale with scene size, not change size.

## What is faked / out of scope

- **Fixed layout.** Each top-level child of the root Node gets a constant
  40x4 rect, stacked vertically. No measurement, no flex, no overflow except
  hard clipping. The spike is about the UPDATE PATH, not layout.
- **Dirty-region mapping is builder-driven.** The owner's dirty set is private
  to package:perception, so each watched box's builder marks its own box index
  on a `RepaintNotifier` (the stand-in for a render object marking its region
  dirty). A real renderer would attach per-element render state. Note the
  pipeline is still honest: even when a box IS repainted, the double buffer is
  what guarantees minimal emission — locality holds at the diff level, not by
  trusting the marks.
- **No input.** Output-only ANSI (cursor moves + SGR + reset); no raw mode, no
  terminal queries, no resize handling.
- **Single-threaded, microtask-driven.** No frame budget/vsync; a flush pass
  runs on the next microtask after the first dirtying event. Sleeps in the
  demo are pacing only.
- **Snapshot literals** (`test/snapshots.dart`) are generated by
  `tool/print_snapshots.dart` and were eyeballed once before locking.
