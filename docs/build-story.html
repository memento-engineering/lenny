01 — the itch"It looked too early."

Watch an LLM drive a web browser long enough and you see the same failure over and over: it clicks something, immediately reads the page, and acts on a half-rendered snapshot. It's racing animations, network calls, partial DOM. A huge slice of agent failures are just "I looked before the app was done reacting."

Flutter is built differently in a way that helps. The framework hands you the frame lifecycle directly. There's a screen-reader-grade semantics tree of the UI. And any package in a debug build can register handlers on the VM service protocol — a real hook into the running app. Put those together and an agent can act, wait until the framework says it's settled, then read a structured snapshot. One turn = one trustworthy observation.

That was the bet. Now it had to get built.

02 — the document that kept rewriting itselfv0.2 → v0.5

May 7, 00:57 — Add initial docs · 02:50 — docs: prd v5

The first cut (v0.2) was a monolith: one binding that knew about routing, state, networking — the works. By v0.5 it had turned itself inside out into a host-and-plugins architecture. The realization that forced it: routing diversity is unsolvable in the core. Flutter has Navigator, Router, go_router, auto_route, beamer, custom RouterConfig — a unified abstraction over all of that is lossy. So don't. Let each router describe itself in its own shape, as a plugin. Same logic for state management, networking, error reporting, telemetry.

The host's job is to be a good plugin host. The plugins' job is to know about specific things.

v0.5 also picked the model target — a local Qwen3.6-35B-A3B served over MLX, with frontier APIs as fallback — and refactored the harness/frontend boundary so the loop is a library consumed by two surfaces: a scriptable CLI and an in-IDE DevTools panel. Same loop. Two faces.

04 — the sprintOne evening. Thirty-one PRs.

May 7, 19:11 — Repo & multi-package scaffolding (#1)

From the first scaffolding commit at 7:11 PM to "PoC complete" the next afternoon, the build ran as a factory: bd held the dependency graph, a supervisor dispatched build agents in parallel across isolated worktrees, each one claimed a ready story, implemented its spec, and a reviewer gated the squash-merge. PRs landed in clusters of three and four because that's how many things were unblocked at once.

In the first ~2½ hours: the binding skeleton, semantics-tree capture with stable IDs, frame-stability signals, the plugin contract + registry, the harness scaffolding + VM-service connection, the exploration_router reference plugin, the model-provider abstraction, exploration_riverpod, Anthropic + OpenAI + swift-infer providers, the JSONL trajectory writer, exploration_dio (which exists partly to prove the busy-state mechanism works), screenshot capture, the plugin authoring guide, and a host diagnostic that warns when interactive widgets ship without semantics.

Then the second wave: the DevTools scaffold with auto-discovery, the runtime-error ring buffer, getStableObservation (the actual stability primitive), the Timeline panel, the core action tools, observation assembly + the per-turn structural diff. A detour to clear CI (analyze warnings, a redundant workflow, perf-tagged tests leaking into the default run). Then memory artifacts + prompt construction with the dynamic merged tool list, the ActionValidator, and — the keystone — the perception-action loop driver with budgets and failure handling.

Then the faces: the DevTools Prompt panel, the Thinking panel (live token streaming), LoopHost wired onto ExplorationSession, and finally the CLI frontend with its AGENTS.md template.

07 — making it visibleThree panels inside DevTools

The CLI is the headless face — CI, batch trajectory collection. The DevTools extension is the one you watch. It auto-discovers whenever an app depends on exploration_flutter, and runs the harness loop inside the panel's own web app — no extra process, no IPC.

Prompt

Type the goal, pick a model, set budgets, hit Start. Later wired to a live plugin manifest from the binding handshake, a provider config block, a dynamic model list pulled from /v1/models, and — eventually — a Start button that actually calls session.run().

Thinking

The model's reasoning trace streams in live (the <think>…</think> content for the Qwen target), then the action it chose and whether it validated. Fastest way to tell if the agent's mental model of your app is coherent.

Timeline

One row per turn — action, one-line diff, running summary. Click to expand into the full observation, reasoning, validation result. Live during the session, browsable after.

…and the plumbing

A build script + commit policy so the compiled web bundle is never committed (CI rebuilds it, no drift), and auto-discovery config shipped from the host package so the tab shows up in standalone DevTools, VS Code, and Android Studio alike.

08 — the boring-important fixesOrdering, contracts, plumbing

  • The binding has to win the race. Some routers (go_router) call WidgetsFlutterBinding.ensureInitialized() deep in their constructor. If app code builds one before the binding installs, the wrong binding wins. The fix: ExplorationBinding.run(ExplorationApp) — claim the slot first, then hand the app a context to build its router/state/client. Discovered, specified, built, merged as lenny-3wt (PR #33).
  • Speak the gateway's language exactly. The swift-infer provider was realigned to match the fs agent wire contract bit-for-bit — Bearer auth, conversation/session/ capture-bodies headers, SWIFT_INFER_AGENT_TOKEN — so one shell export and one inspection toolchain serve both clients.
  • Defer the DevTools reads. A serviceManager read above DevToolsExtension threw on the first frame; the fix moved those reads below it. Small bug, real crash, lenny-g4u.
  • Pre-launch rebuild. Rebuild the DevTools extension bundle, gitignore the CLI's trajectory output, and the stack is presentable.
